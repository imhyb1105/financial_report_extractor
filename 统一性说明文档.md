# 统一性说明文档（财税小工具）

版本：v1.1.0
更新日期：2026-03-15
适用范围：增值税计算器、附加税费计算器、企业所得税计算器，以及后续新增的所有财税小工具。

---

## 目标与原则
- 降低维护成本：统一技术栈、视觉规范与交互方式。
- 提升一致体验：统一数字格式、错误提示与政策可溯源性。
- 保持计算独立：各工具计算逻辑独立，但呈现与治理方式一致。

## 当前差异速览（背景）
- 技术依赖：增值税计算器使用 Bootstrap 5；附加税费计算器为原生自定义样式；企业所得税计算器使用 Bootstrap 5 分步向导模式。
- UI结构：增值税为多标签页（一般/小规模/混合销售），含差额与明细表；附加税费为单页表单 + 结果网格；企业所得税为5步向导模式。
- 字体与色板：增值税固定 `Microsoft YaHei`、蓝色系；附加税费采用系统字体族与 CSS 变量色板。
- 数值与提示：增值税结果使用千分位与 `alert()` 错误提示；附加税费统一两位小数并内联错误区；企业所得税使用千分位 + 内联错误区。
- 配置治理：增值税税率硬编码；附加税费使用独立 `rates.config.json` 并展示版本/生效日期；企业所得税使用 `cit.config.json` 完整配置。
- 知识库联动：附加税费页面提供知识库链接；增值税页面未显式提供入口；企业所得税提供完整知识库链接。

---

## 统一性总则（决策）
- 技术栈：默认采用"原生 HTML + 自定义 CSS + 原生 JS"。如确需第三方库（如 Bootstrap），须在所有工具中一致启用并统一组件用法。
- 数字格式：结果区统一"千分位 + 两位小数"。内部计算保留精度，最终输出统一格式化。
- 校验与提示：统一使用表单内联错误区（`aria-live="polite"`），避免 `alert()` 作为主要错误呈现。
- 配置治理：各工具的税率/政策口径集中在独立 `config` 文件，页面统一展示"版本/生效日期"与来源（PRD + 知识库）。

## 税种复杂度与计算器模式

根据税种计算复杂度，采用不同的计算器模板：

### 复杂度判定规则

| 复杂度 | 税种 | 计算器模式 | 知识库文件数 | 核心特征 |
|--------|------|-----------|-------------|----------|
| **简单** | 印花税、契税、附加税、车船税、车辆购置税、关税、城建税、教育费附加 | 单页简单表单 | 5个 | 税基明确、税率固定、无复杂调整 |
| **中等** | 房产税、土地增值税、消费税、城镇土地使用税、资源税、环境保护税、耕地占用税 | 单页增强表单（Tab切换） | 5个 | 多档税率、需判断适用情形 |
| **复杂** | **企业所得税**、**个人所得税**、**增值税** | **分步向导模式（5步）** | **6个** | 应纳税所得额计算复杂、多项调整、分档累进 |

### 复杂税种核心难点

#### 企业所得税：应纳税所得额计算
```
应纳税所得额 = 会计利润总额
            + 限额扣除超限调增（业务招待费、广告费等）
            + 不得扣除项目调增（罚款、滞纳金等）
            - 免税/不征税收入调减（国债利息、股息红利等）
            - 加计扣除（研发费用100%、残疾人工资100%）
            - 亏损弥补
```

#### 个人所得税：综合所得计算
```
应纳税所得额 = 收入合计（工资+劳务+稿酬+特许权使用费）
            - 减除费用（60000元/年）
            - 专项扣除（社保公积金）
            - 专项附加扣除（子女教育、住房贷款等）
            - 其他扣除

应纳税额 = 应纳税所得额 × 适用税率 - 速算扣除数
```

#### 增值税：进项抵扣与多档税率
```
应纳税额 = 销项税额 - （进项税额 - 进项转出）
```

### 复杂税种特殊规范

- **采用5步向导模式**：基础数据 → 限额扣除 → 纳税调整 → 优惠扣除 → 结果
- **知识库额外增加** `5_{税种名称}纳税调整详解.md`
- **配置文件需包含**完整的扣除限额、加计扣除等配置
- **自动计算限额扣除**：业务招待费（Min(60%, 5‰)）、广告费（15%）等

## 技术栈与依赖治理
- 默认不依赖外部 CDN；如确需依赖（例如图标或框架），需：
  - 在所有工具中一致使用；
  - 明确降级方案与离线可用性；
  - 记录版本与来源于配置或 README。
- 共享基础资源建议（已落地到 `common/`）：
  - `common/styles.css`：色板、卡片、按钮、表单、栅格的统一样式。
  - `common/components/utils.js`：数字格式化、两位小数、复制与校验工具方法。
  - `common/components/version.js`：配置驱动的版本与生效日期展示。
  - `common/components/export.js`：结果区打印导出（PDF）。

## UI 与视觉统一
- 字体族统一：
  - `system-ui, -apple-system, Segoe UI, Roboto, Helvetica, Arial, PingFang SC, Microsoft YaHei, sans-serif`
- 色板统一（CSS 变量示例）：
  - `--bg: #f7f7fa`、`--card: #ffffff`、`--text: #1f2937`、`--muted: #6b7280`
  - `--primary: #2563eb`、`--primary-contrast: #ffffff`、`--danger: #dc2626`
- 组件规范：
  - 卡片：圆角 10px、轻阴影 `0 1px 2px rgba(0,0,0,0.06)`、内边距 16px。
  - 按钮：主次态（`btn-primary` / `btn-secondary`），悬浮轻微亮度变化。
  - 表单：统一焦点态（描边 + 阴影）、对齐与间距一致。
- 导航一致性：选择一种方案并贯彻：
  - 方案 A：单页 + 卡片分区（推荐，轻量一致）—— 适用于简单/中等税种。
  - 方案 B：标签页（tabs）分区（如业务复杂确需分区）—— 适用于中等税种。
  - 方案 C：分步向导模式（5步）—— 适用于复杂税种（企业所得税、个人所得税、增值税）。

## 表单与校验统一
- 关键输入统一加 `required` 与非负校验，避免负值。
- 错误呈现：
  - 使用内联错误区：`<div class="errors" aria-live="polite"></div>`。
  - 错误文案简洁、可定位，避免弹窗中断流程。
- 可用性提示统一在结果注释区域（`note`），避免混用不同风格提示框。

## 数字与格式统一
- 统一函数语义：
  - 输入解析：`parseAmount(input)`，返回数字或 `NaN`。
  - 两位小数：`toFixed2(n)`，四舍五入至两位。
  - 结果格式：`formatMoney(n)`，千分位 + 两位小数。
- 中间计算使用原始数值；渲染阶段统一格式化。

### 生效日期规则（统一）
- 展示语义：页面顶部与页尾显示"最后生效日期"，以相关法律、法规、条例或规定的最后生效日期为准。
- 字段优先级（从高到低）：
  - `effective_until`
  - `effective_period.to` 或 `effective_period.end`
  - `effective_date`（单值即视为最后生效日期）
- 配置示例（可选生效区间）：
```json
{
  "version": "2026.03-001",
  "effective_period": { "from": "2023-01-01", "to": "2027-12-31" }
}
```
- 展示规范：若提供生效区间，顶部仍以"最后生效日期"展示；需要完整区间显示时，可在结果说明或页脚补充 `from ~ to` 文案。

### 版本号格式（统一）
- 格式：`YYYY.MM-NNN`
- 示例：`2026.03-001`、`2026.03-002`
- 说明：`YYYY.MM` 为更新年月，`NNN` 为当月修订序号

## 术语约定
- 差额征税：按规则对销售额进行差额扣除后计税。
- 混合销售：多税目合并销售的计税规则，分别核算/从高适用税率两模式。
- 留抵税额：当期形成的进项大于销项的余额，按政策结转或退税。
- 应纳税所得额：经过纳税调整后的计税基数，不同于会计利润。
- 限额扣除：税法规定扣除上限的项目（如业务招待费、广告费）。
- 加计扣除：在实扣基础上允许额外扣除的优惠政策（如研发费用100%加计扣除）。
